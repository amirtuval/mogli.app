# Mogly — Design & Build Specification

## Overview

**Mogly** is a native desktop app (macOS, Windows, Linux) that provides a unified inbox and calendar view across multiple Google accounts. It uses the Gmail REST API and Google Calendar API — no IMAP, no local email indexing. Full message bodies are fetched on demand. The app is built with Tauri v2 (Rust backend) + React + TypeScript/TSX frontend.

**Core principles:**
- Never download or index full email bodies locally
- All Google API calls go through the Rust backend via Tauri commands
- Frontend is TypeScript/TSX throughout — no plain JS
- Rust/TS type boundary is managed by `tauri-specta` (auto-generated bindings)
- Account tokens stored in OS keychain via `keyring` crate
- No SQLite — account config and sync state stored as JSON in Tauri's `app_data_dir`
- State management on frontend: TanStack Query for server state, Zustand for UI state
- **No UI element is built until the feature behind it is implemented** — no placeholder buttons

## Tech Stack

### Frontend
- React 18 + TypeScript (TSX)
- Vite (bundler, Tauri default)
- TanStack Query v5 — async data fetching, caching, background refetch
- Zustand — UI state (selected email, active view, theme, active accounts)
- `date-fns` — date formatting
- Fonts: IBM Plex Sans + IBM Plex Mono (Google Fonts)

### Styling
- **CSS Modules** — one `.module.css` file per component, scoped class names, zero runtime
- **CSS custom properties** — all theme tokens (colors, borders, text) as variables on `document.documentElement`
- Vite supports CSS Modules natively, no additional config required
- Dynamic theme switching: update CSS vars on root element via JavaScript, all components react instantly
- Hover, focus, transitions written in plain CSS within module files — no JS workarounds needed
- Inline styles used only for truly dynamic per-instance values (e.g. account color on a specific element)

Pattern:
```tsx
// EmailList.tsx
import styles from './EmailList.module.css'
<div className={`${styles.row} ${email.unread ? styles.unread : ''}`}>...</div>
```
```css
/* EmailList.module.css */
.row { border-left: 2.5px solid var(--account-color); transition: background 0.1s; }
.row:hover { background: var(--bg-hover); cursor: pointer; }
.row.unread .sender { font-weight: 500; color: var(--text-primary); }
```

### Backend (Rust / src-tauri)
- Tauri v2
- `tauri-specta` + `specta` — auto-generate TypeScript bindings from Rust command signatures
- `reqwest` + `tokio` — async HTTP client for Google API calls
- `keyring` — OS keychain for OAuth token storage
- `oauth2` — OAuth2 flow
- `serde` + `serde_json` — serialization
- `tauri-plugin-notification` — OS notifications
- `tauri-plugin-store` — JSON file storage for account config + sync state

### Testing
- **Rust unit tests**: `#[cfg(test)]` modules in each `.rs` file, run with `cargo test`
- **Rust integration tests**: `src-tauri/tests/` directory, test full command flows with mocked HTTP
- **HTTP mocking**: `mockito` crate — mock Google API responses in Rust tests
- **Frontend unit tests**: Vitest + React Testing Library
- **Frontend component tests**: Vitest + React Testing Library, mock `commands` from bindings
- **No E2E tests in phases 1–5** — Tauri E2E (via WebDriver) is complex to set up; defer until post-v1

### Google APIs used
- Gmail API v1
- Google Calendar API v3
- OAuth2 scopes required:
  - `https://www.googleapis.com/auth/gmail.modify`
  - `https://www.googleapis.com/auth/calendar`

## OAuth Flow (Desktop)

Google OAuth in a desktop app uses a **loopback redirect** (127.0.0.1) rather than a custom URI scheme, which Google explicitly supports for installed apps.

Flow:
1. App starts a temporary local HTTP server on a random port (e.g. `127.0.0.1:PORT`)
2. Opens system browser to Google's OAuth consent URL with `redirect_uri=http://127.0.0.1:PORT`
3. User consents in browser
4. Google redirects to `127.0.0.1:PORT?code=AUTH_CODE`
5. Local server captures the auth code
6. App exchanges code for access + refresh tokens
7. Tokens stored in OS keychain under key `mogly::{account_email}`
8. Local server shuts down

Token refresh: before every API call, check expiry. If expired, use refresh token to get new access token silently.

## Sync Strategy

- On app launch: fetch fresh message metadata for all accounts
- Background polling every 2 minutes via a Tokio interval task
- Use Gmail `history.list` with stored `historyId` to fetch only changes since last sync
- On new messages detected: emit Tauri event to frontend → TanStack Query invalidates → refetch
- Frontend listens with `listen('mail:new', ...)` from `@tauri-apps/api/event`

## Testing Philosophy

### What to test and why

**Rust unit tests** — test pure logic in isolation:
- OAuth token parsing and expiry checking
- Gmail API response parsing (headers, body decoding, MIME part traversal)
- Calendar event parsing (dateTime vs date, timezone handling)
- Message merge + sort logic (multi-account unified inbox ordering)
- `historyId` update logic

**Rust integration tests** (with `mockito`) — test full command flows against mocked HTTP:
- `get_messages` returns correctly shaped `Vec<MessageMeta>` from a mocked Gmail response
- `get_thread` correctly decodes base64url body from mocked response
- Token refresh is triggered when access token is expired
- `get_events` returns events from multiple mocked calendars

**Frontend unit tests** (Vitest) — test pure logic:
- `uiStore`: state transitions (toggle account, change view, select thread)
- `theme.ts`: CSS var application
- Date/time formatting helpers
- Multi-account event merge and sort

**Frontend component tests** (Vitest + React Testing Library) — test components with mocked Tauri commands:
- Mock `commands` module: `vi.mock('../types/bindings', () => ({ commands: { ... } }))`
- `EmailList`: renders correct number of rows, unread styling, account color borders
- `EmailDetail`: shows subject/from/body, archive button calls correct command
- `CalendarView`: events positioned at correct times, today column highlighted
- `MiniCal`: correct days rendered, today highlighted, month navigation works
- `WelcomePage`: "Connect" button calls `add_account`, account appears after success

**What NOT to test in phases 1–5:**
- Full E2E with real Google accounts (too slow, requires credentials in CI)
- Tauri WebDriver E2E (complex setup, low ROI at this stage)
- Visual regression tests (defer until UI is stable)

## Phase 1 — Scaffold + OAuth + Debug View

**Goal:** A running Tauri app that can authenticate with Google and display raw API data. No real UI yet. Validates the highest-risk pieces (OAuth, token storage, Google API access) before any UI investment.

### Tasks

#### 1.1 Project scaffold
- `tauri init` with Vite + React + TypeScript template
- Install frontend deps: `react`, `react-dom`, `@tanstack/react-query`, `zustand`, `date-fns`, `vitest`, `@testing-library/react`, `@testing-library/user-event`
- Install Tauri plugins: `tauri-plugin-store`, `tauri-plugin-notification`
- Install Rust deps in `Cargo.toml`: `tauri-specta`, `specta`, `reqwest`, `tokio`, `serde`, `serde_json`, `keyring`, `oauth2`, `mockito` (dev), `uuid`
- Configure `tauri-specta` to export bindings to `src/types/bindings.ts`
- Set up `tsconfig.json` with strict mode
- Set up `vite.config.ts` with Vitest config
- Set up `.github/workflows/ci.yml` (lint, fmt, test jobs as described above)
- Set up `.github/workflows/release.yml` (tauri-action matrix build)
- Set up ESLint + Prettier configs
- Set up `rustfmt.toml` and Clippy config

#### 1.2 OAuth implementation (Rust)
- Implement `google/oauth.rs`:
  - Start local HTTP server on random available port (`std::net::TcpListener`)
  - Build Google OAuth2 authorization URL with scopes: `gmail.modify`, `calendar`
  - Open URL in system browser via Tauri shell API
  - Wait for redirect to `http://127.0.0.1:PORT?code=...`
  - Exchange code for access + refresh tokens using `oauth2` crate
  - Store tokens in OS keychain: key `mogly::{email}`
  - Return `Account` struct
- Implement `refresh_token_if_expired(email: &str) -> Result<String>`:
  - Check expiry, if expired exchange refresh token for new access token
  - Update stored token in keychain

#### 1.3 Account management commands (Rust)
- `commands/auth.rs`:
  - `add_account()` → triggers OAuth flow, saves account to JSON store, returns `Account`
  - `remove_account(account_id)` → removes from store + keychain
  - `list_accounts()` → reads from `app_data_dir()/accounts.json`, returns `Vec<Account>`
- `store.rs`: read/write `accounts.json` via `tauri-plugin-store`

#### 1.4 Debug view (TSX)
- Minimal React app — intentionally unstyled
- On load: call `commands.listAccounts()`
- If no accounts: "Add Account" button → calls `commands.addAccount()`
- If accounts exist: render raw JSON of accounts list
- "Fetch messages" button → calls `commands.getMessages([accountId], 'INBOX', null)`
- Display raw JSON response
- This view is removed entirely in Phase 2

#### 1.5 tauri-specta bindings
- All commands registered and exported
- Run binding generation, verify `src/types/bindings.ts` is correct
- Commit generated bindings to repo
